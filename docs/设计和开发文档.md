# 第一章 需求分析

## 1.1 开发背景

在项目立项阶段，我们发现两个问题比较典型：一类用户有明确的页面或应用想法，但缺少完整的开发能力，往往停留在“能描述需求、不能落地成果”的阶段；另一类用户虽然能借助通用 AI 工具生成部分代码，但生成结果常常分散、难预览、难继续修改，更难形成可部署、可传播、可协作的完整产物。基于这一现实问题，本团队将织码睿奇定位为“面向数字产物创作全过程的智能开发平台”。

## 1.2 对标对象与作品定位

我们希望开发一款面向全社会的 AI 驱动平台，通过调用 AI 大模型能力，实现用户创意、工作业务与日常需求向可执行数字产物的快速转化，提供需求分析、素材选配、代码生成、可视化编辑、项目部署及分享传播的一体化解决方案。调研后，我们发现和同类产品的区别：

| 比较维度 | 通用 AI 代码助手 | 织码睿奇 |
| --- | --- | --- |
| 主要输入方式 | 文本指令、代码补全 | 自然语言 + 优化输入 + 提示词模板 + 多模式选择 |
| 主要输出形态 | 代码片段或局部工程 | 单页代码、多文件代码、Vue 工程框架 |
| 修改方式 | 继续提问或手工改代码 | 对话修改 + 可视化编辑 + 构建校验 |
| 落地能力 | 往往需要自行收尾 | 已形成预览、构建、部署闭环 |
| 传播与互动 | 一般较弱 | 支持点赞、收藏、评论、分享、社区 |
| 面向人群 | 开发者 | 同时面向普通用户与企业团队用户 |

## 1.3 目标用户与核心功能、性能需求

我们将目标用户归纳为三类：**个人用户**、**商户用户**、**企业或团队用户**

围绕这些用户，目前系统已经实现的核心功能包括：**AI 生成**、**多种结果形态**、**流式交互**、**质量增强**、**工程落地**、**传播协作**、**使用答疑**

主要性能要求包括：**异常可恢复**、**结果可运行**、**平台可扩展**、**并发下数据稳定**

综上，本团队希望解决的并不是单一“**生成页面**”的问题，而是数字产物从生成到落地再到社区讨论、传播的完整使用问题。


# 第二章 详细设计

## 2.1 后端设计

### **3.1.1 平台化后端架构设计**

本项目后端采用多模块架构：

一方面，将用户、生成、社交、提示词、文件等核心业务拆解为**边界清晰的独立模块**；另一方面，又通过统一模型、公共常量与内部服务接口**保持整体一致性**。

我们这么设计是为了系统既能支撑**生成数字产物**生成这种高复杂度的主要业务，又能不断扩展**协作、社区、客服、知识库**等外围能力，而不会让全部逻辑堆积在单体架构的工程中，不是“功能堆砌”，而是面向真实企业级开发。

### **2.1.2 双模式 AI 生成架构设计**

本项目在 AI 生成链路上没有采用单一方式，而是考虑设计成了“工作流模式 + Agent 模式”的双引擎方案。

**工作流模式：**流程可控、阶段清晰、结果稳定，适合结构相对明确的生成任务；

**Agent 模式：**后者强调自主规划、工具协同和复杂任务处理能力，更适合工程化程度更高的应用生成场景。

**双模式并存的价值在于：**它不是简单增加一个“模式切换按钮”，我们将“稳定性”和“灵活性”同时纳入系统设计，使平台**既能承担规范化生成任务**，**也能承接更具探索性的复杂创作请求**，这正是本项目在 AI 应用设计上的一个重要亮点。

### **2.1.3 增强生成产物品质的设计**

后端采用了**分层增强思路**，将**系统提示词**、**行业模板**、**历史上下文**和**知识库检索**互相组织起来，提升AI生成质量。

我们这样做的核心目的并不是单次生成更长的文本，而是让模型更理解**用户要求的业务场景**、**更理解用户意图**，也更**理解不同代码类型之间的差异**。

这种设计的**创新价值**主要体现在三个方面：

- **针对不同场景生成**：系统能够根据**生成类型、用户身份和具体业务场景**切换不同的引导策略，**降低“答非所问”和“产物风格”的 AI 幻觉概率**。
- **产物构建的连续性**：针对同一种 AI 产物，引入**产物构建历史**上下文，使系统不再把二次产物构建都视为孤立请求，而是能够围绕同一产物持续迭代。
- **AI 调教经验可积累**：知识库、系统提示词采用可视化管理方式，后续可持续补充平台经验、规范样例和业务知识，使系统能力**具备长期沉淀、学习、自我增强的空间**。

**总之 —— 本项目并非直接调用大模型接口，而是通过多层代码组织、工具构建、流程编排、用户引导等方式，把通用模型转化为更贴合数字产物创作场景的“专家”。**

### **2.1.4 长任务流式交互设计**

项目后端采用了**流式会话**与**断点续接**相结合的设计思路，把**“任务执行过程”**和**“当前连接状态”**适度解耦。

我们团队这样做的意义在于，用户看到的不再只是最终结果，而是一个**可持续观察、可中途恢复、可重复跟随**的生成过程。对于 AI 产品来说，这种设计可以提升**交互的稳定性**和**用户主观感受上对AI产物的可信度**，也能应对本项目**部署上线后真实使用环境下的网络不确定性**。

这一机制的创新点在于，我们不再是单纯使用 SSE 返回内容，而是进一步考虑了**长任务生命周期管理、事件缓存、重连跟随和状态恢复**等问题，确保系统具备了**比普通对话式生成器更强的连续服务能力**。

### **2.1.5 产物工程化落地设计**

本项目在 AI 构建代码完毕后，会开始进行“**产物落地**”。也就是说，我们关注的不只是 AI 会不会生成，还要可以包括“**能不能预览、能不能构建、能不能部署、能不能继续迭代**”这四大后续问题。

针对不同类型产物，后端采用差异化处理思路：**轻量页面**强调快速预览与下载，**多文件项目**强调结构完整性，而 **Vue 工程**则进一步强调依赖处理、构建验证与部署发布。

因此，我们完成了一条较为完整的“**生成—校验—构建—发布**”链路，这也是区别于普通网页生成 Demo 的关键技术。

### **2.1.6 用户、多人协作与社区关系网设计**

后端不仅支持基本身份管理，还将**好友关系、即时沟通、协作邀请、消息通知**等机制嵌入数字产物创作场景之中，使“个人创作”能够扩展为“**多人社区协同的创作**”。

简单来说，用户既可以独立完成产物生成，也可以围绕某个产物进行**沟通、转发、协作与共同完善**。

### **2.1.7 社交传播与社区运营设计**

本项目将**点赞、收藏、评论、分享、热门排行和社区交流**作为后端能力体系中的重要一环，因为平台设计目标并不局限于“生产内容”，而是进一步考虑“**内容如何传播、完善和被再次利用**”。

在设计层面，社交模块体现出两个明显特点：

- **社区驱动产物生成的热度**：用户生成产物后，可以围绕产物形成点赞、收藏、评论、分享等反馈，促进优质内容扩散。
- **让社区具备运营能力**：热门排行、社区置顶等机制使平台具备基础内容运营能力，有利于优质 AI 产物的积累和站内活跃氛围形成。

同时，我们对高频互动场景下的**数据一致性问题、高流量抗压能力**也进行了专门设计，通过 Redis 缓存实现滑动窗口限流使社区模块在平台规模扩大后仍具备较好的稳定性基础。

### **2.1.8 AI 客服服务化设计**

AI 客服在本项目中不是边缘附属功能，通过 RAG 知识库深入学习了本系统的所有详细功能与能力边界，作为首页常驻功能围绕平台使用过程中遇到的所有问题提供详细的解答。

## 2.2 UI 设计

### **2.2.1 整体界面风格设计**

在界面设计上，本项目**没有采用过度炫技、俗套的的 UI 设计**，而是选择了**更贴近生产工具与互联网大厂平台同款的简洁风格**，**将“入口页”和“创作页”组织在同一工作台里**。

用户进入系统后，不需要在多个页面之间频繁跳转，而是在同一工作台内中完成所有的需求输入、产物选择、过程追问、结果预览和后续操作。

**用专业点的话来说，就是降低交互按钮不集中带来的那种割裂感，保持用户创作时的“心流”。**

## 2.4 本项目设计难点与解决方案

### **2.4.1 AI 生成模式单一，难以同时兼顾稳定性与灵活性**

**面临问题：**

很多 AI 生成系统只有一套固定模式：要么流程过于死板，面对复杂需求时不够灵活；要么完全交给模型自由发挥，生成过程又容易失控。对于本项目这类“既要快速出结果，又要支持继续修改和工程化落地”的平台来说，单一模式很难同时满足简单任务和复杂任务两类场景。

**解决方式：**

本团队在生成主链路中采用了“**双模式协同**”设计。

- 一方面，基于 LangGraph4j 构建节点式工作流生成链路，将资源收集、提示优化、类型决策、代码生成、质量审查、项目构建等关键环节组织成严谨流程，突出**“稳定”**；
- 另一方面，引入 Agent 自主规划模式，让 AI 能结合工具和上下文自主拆解任务、连续推进，突出**“自主”**。

我们这种设计带来的直观能力是：平台既可以像一条标准化生产线，严格约束 AI ，适合稳定输出；又可以作为一个能够自主思考的 Agent 助手，适合处理更复杂、更开放的创作任务。

### **2.4.2 多用户同时在线时，长耗时环节阻塞容易降低用户整体生成体验**

**面临问题：**

在真实平台场景中，AI 生成并不只是一次文本返回，还会伴随**代码生成、思考决策、生成任务列表、网络资源搜索、图片生成、页面预览准备、产物封面截图**等等长耗时环节。如果这些任务全部堵在主流程上，一旦多人同时在线使用，系统就容易出现“**请求排队、界面转圈、反馈变慢**”的问题，直接影响体验。

**解决方式：**

针对这一问题，本项目将这类典型长耗时操作转入 JDK 21 **虚拟线程中异步执行**，把慢任务从主交互链路中适度剥离。这样做的好处是，系统在面对多用户并发和多步骤生成时，不必用大量传统线程硬扛，也不会让单个慢操作把整体体验拖住。

我们的这一设计让平台在“多人同时生成、多任务并行推进”的情况下仍能保持较好的响应，线上有很强的并发性能。

### **2.4.3 高频多轮对话下，AI 服务实例反复创建会造成额外资源消耗**

**面临问题：**

数字产物创作往往不是一次问答就结束，而是会经历多轮追问、多轮修改和持续优化。如果每次对话都重新创建完整的 AI 服务对象、重新装载历史上下文和工具配置，就会给系统带来较大的资源消耗，也会影响连续对话的稳定性。

**解决方式：**

本项目将工作流生成、Agent 生成和 AI 客服等重量级 AI 服务实例按产物或会话维度存入 **Caffeine 本地缓存**，并结合 **Redis 对话记忆与历史记录恢复机制**，实现“**同一任务尽量复用、不同任务保持隔离**”的服务缓存方式。这样既**减少了高频场景下反复创建对象的开销**，也**降低了重复装载历史上下文带来的压力**。

### **2.4.4 AI 调用过程容易变成“黑箱”，管理员难以及时掌握系统运行状态**

**面临问题：**

AI 系统一旦进入实际运行阶段，其实我们最怕的并不是单次回答不理想，而是完全看不清系统**到底用了多少次模型、消耗了多少 Token、响应是快是慢、哪些环节最容易出错**。如果缺少实时监控能力，平台就很难持续优化，也不利于后续运维和展示。

**解决方式：**

本项目在 AI 调用链中引入监听机制，**对请求次数、错误次数、Token 消耗和响应时长等关键指标进行采集**，并通过 Spring Boot Actuator 暴露 Prometheus 采集端点；同时，项目中已经提供了 Prometheus 抓取配置、Grafana 部署文档和监控看板文件，便于形成可视化监控体系。为适应并发场景，系统还使用 ConcurrentHashMap 对计数器与计时器进行缓存管理，避免重复创建监控对象。

使系统具备了“能看见 AI、能分析 AI、能持续优化 AI”的**可视化管理能力**。

### **2.4.5 HOT 接口在高频访问下容易出现瞬时压力**

**面临问题：**

生成入口、查询入口、热点业务接口一旦被频繁访问，很容易在短时间内形成请求高峰。如果完全放开，不仅会影响单个用户体验，还可能把整个平台拖入短时拥堵状态。尤其是 AI 相关接口，本身成本更高，更需要提前做好保护。

**解决方式：**

本项目对核心接口引入了基于 **Redisson 的分布式限流机制**，可**按用户、IP、接口等多个维度实施保护**，避免高频请求直接冲击主服务。同时，**系统将对话记忆等高频上下文数据交由 Redis 支撑**，减少重复加载和重复计算带来的额外压力。

简单来说，相当于给平台加上了“高峰保护阀”和“热点缓冲层”。

### **2.4.6 RAG 知识文档难以直接转化为 AI 可理解、可复用的知识能力**

**面临问题：**

很多系统虽然积累了大量开发经验、规范文档和平台说明，但这些内容往往只是“存着”，并不能真正参与 AI 生成。这会带来一个非常直观的问题：**我们团队在观测平台在创作 AI 产物后，明明总结了经验，却无法在把这些经验稳定地用到下一次的生成结果中。**

**解决方式：**

本项目直接设计了一套**本地 RAG 知识库系统**，通过将知识库落库处理来动态地管理 RAG 知识库——

- **1.文档收集与切割：**使用 LangChain4j 提供的文档处理与切分能力，**对知识文档进行整理和分段**，尽量保留语义完整性。
- **2.向量转换与存储：**采用阿里云 **text-embedding-v4 向量模型**，将文本片段批量转为向量，并摄入 **PostgreSQL 的 pgvector 向量数据库中**。
- **3.文档过滤与检索：**通过业务类型、生成类型等**元数据对知识片段进行区分**，让不同模式只检索与自己相关的知识，减少“拿错资料”的问题。
- **4.查询增强与内容关联：**在对话服务构建阶段接入检索增强器，通过**内容检索器与内容注入器**，把命中的知识片段自动补充到用户问题上下文中。

这样一来，管理员可直接一键向量化pgvector 到向量数据库，方便**把开发团队积累的 AI 生成经验真正转化为持续可用、持续完善的 AI 知识库，**这也是本项目相较普通问答式 AI 项目更有优势的一点。

### **2.4.7 长时间生成任务容易因刷新页面或网络波动而中断**

**面临问题：**

AI 生成、工程构建和客服问答都不是瞬间完成的任务。用户在等待过程中，如果刷新页面、切换网络或短暂断线，就很容易出现“前面白等了、结果丢了、必须重来一遍”的情况。这种中断感会严重破坏使用体验。

**解决方式：**

为了解决这一问题，本项目在流式输出时采用了“**任务执行与当前连接解耦**”的思路：服务端持续缓存流式事件，客户端可基于事件位置继续跟随后续内容，从而实现断线续接和页面重连恢复的**健壮连接**。

### **2.4.8 生成结果如果停留在代码层面，普通用户难以应用到工作、生活**

**面临问题：**

如果系统只是输出一段代码，用户还需要自己安装依赖、执行构建、检查产物、处理预览和部署问题，那么“会生成”与“能交付”之间仍然隔着很长一段距离。

**解决方式：**

本项目把构建、预览和部署添加到 AI 生成链的后半段，形成**“从需求到可运行产物”**的完整业务。尤其是在 VUE 生成的场景中，需要进一步校验项目结构、执行依赖安装、完成项目构建并检查输出结果，让 AI 生成真正向可展示、可运行、可发布的成果迈进一步。

## 5.2 Linux 服务器部署步骤

### **5.2.1 上传文件与目录规划**

下面按照如下项目路径给出示例指令：

mkdir -p /opt/RichCodeWeaver/{server,web,middleware,logs}

- server/：存放 6 个后端 Jar；
- web/：存放前端解压后的静态文件；
- middleware/：存放 Nacos、Prometheus、Grafana 等中间件；
- logs/：存放运行日志。

### **5.2.2 数据库初始化**

在 MySQL 中创建业务数据库后，执行项目提供的 sql/RichCodeWeaverDB_MySQL.sql，完成核心表和初始化数据导入。

### **5.2.3 启动中间件**

推荐按“Redis → Nacos → Prometheus → Grafana”的顺序启动：

\# Nacos（单机模式）  
cd /opt/RichCodeWeaver/middleware/nacos/bin  
sh startup.sh -m standalone  

\# Prometheus  
cd /opt/RichCodeWeaver/middleware/prometheus-3.11.2.linux-amd64  
nohup ./prometheus --config.file=prometheus.yml > /opt/RichCodeWeaver/logs/prometheus.log 2>&1 &  

\# Grafana  
cd /opt/RichCodeWeaver/middleware/grafana-12.1.0  
nohup ./bin/grafana-server web > /opt/RichCodeWeaver/logs/grafana.log 2>&1 &

其中，Prometheus 只需把抓取目标配置为 localhost:8101/api/actuator/prometheus，即可采集生成服务的核心指标。

### **5.2.4 后端配置与启动**

6 个 Jar 均按生产环境方式运行，重点补齐以下配置：MySQL、Redis、Nacos、邮箱、OSS、AI 模型服务，以及实际使用到的 RAG 配置。配置完成后，可按如下顺序启动：

nohup java -jar /opt/RichCodeWeaver/server/rich-code-weaver-generator-1.0.jar --spring.profiles.active=prod > /opt/RichCodeWeaver/logs/generator.log 2>&1 &  
nohup java -jar /opt/RichCodeWeaver/server/rich-code-weaver-user-1.0.jar --spring.profiles.active=prod > /opt/RichCodeWeaver/logs/user.log 2>&1 &  
nohup java -jar /opt/RichCodeWeaver/server/rich-code-weaver-file-1.0.jar --spring.profiles.active=prod > /opt/RichCodeWeaver/logs/file.log 2>&1 &  
nohup java -jar /opt/RichCodeWeaver/server/rich-code-weaver-social-1.0.jar --spring.profiles.active=prod > /opt/RichCodeWeaver/logs/social.log 2>&1 &  
nohup java -jar /opt/RichCodeWeaver/server/rich-code-weaver-prompt-1.0.jar --spring.profiles.active=prod > /opt/RichCodeWeaver/logs/prompt.log 2>&1 &  
nohup java -jar /opt/RichCodeWeaver/server/rich-code-weaver-server-admin-0.0.1.jar --spring.profiles.active=prod > /opt/RichCodeWeaver/logs/admin.log 2>&1 &

对应端口：8101（generator）、8102（user）、8103（file）、8104（social）、8105（prompt）、9001（admin）。

### **5.2.5 前端部署与 Nginx 代理**

将 rich-code-weaver-web.zip 解压到 /opt/RichCodeWeaver/web，再由 Nginx 提供静态资源与反向代理。

Nginx 配置模板详见源码中的 “\\RichCodeWeaver\\bat\\nginx.conf”,内含详细的路由转发、静态资源兼容性配置。

# 第六章 项目总结

## 6.1 制作过程中的体会

本项目中，我们团队最大的体会，不在于单点功能的实现，而在于如何将一个想法拆解成可落地、可协作、能持续迭代的完整作品。开发期间，本团队将任务拆分为主链路生成、用户与权限管理、部署与预览、社区互动、客服支持、后台管理等多个相对独立的模块，优先确保主流程顺畅运行，再逐步完善各项配套能力，这种方式不仅降低了开发风险，也有效提升了团队协同效率。

项目推进过程中，本团队遇到的突出困难主要有三类：

- 一是AI生成结果不够稳定，本团队通过优化提示词、完善工作流、强化记忆与知识增强等方式不断调试优化；
- 二是长任务场景下，生成、预览、部署、重连各环节易出现链路断裂问题，需要反复联调排查；
- 三是随着功能不断增加，前后端接口、页面交互与数据结构之间的协调成本大幅上升。

在解决这些问题的过程中，本团队对“先拆解、再联调、边验证边收敛”的软件工程方法有了更深刻的理解，同时也提升了系统设计、接口协同及问题排查的综合能力。

## 6.2 后续升级与推广设想

后续升级方面，本团队计划从稳定性、标准化、实用化三个方向继续推进项目优化：

- 一是完善自动化测试、统一部署脚本及监控告警能力，提升项目工程成熟度；
- 二是持续优化AI生成质量、模板推荐及知识库管理功能，进一步降低用户使用门槛；
- 三是深耕社区、协作及客服能力，让平台不仅能实现“高效生成”，更能满足用户“持续便捷使用”的需求。

从商业推广层面来说，本项目适合对口各个行业的企业提效、商户宣传、个人工具等需求场景。核心价值不在于替代完整的软件开发流程，而在于大幅缩短“从想法到可演示原型”的时间成本。

总体而言，通过本次开发，本团队更加深刻地认识到：**优秀的作品，不仅需要扎实的技术实现，更要重视创新思维、任务拆解、团队协同、用户体验以及后续的系统优化与维护**。
