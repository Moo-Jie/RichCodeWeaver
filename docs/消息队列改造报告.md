# 织码睿奇消息队列改造报告

## 版本信息
- 项目名称: 织码睿奇 (RichCodeWeaver)
- 版本编号: 2.0.0
- 报告日期: 2026-05-06
- 编制目的: 基于流量控制、异步解耦、削峰填谷的企业级消息队列改造方案

---

## 一、现状分析

### 1.1 系统架构概述

本系统采用多模块微服务架构,核心模块包括:
- rich-code-weaver-generator: AI代码生成核心服务(端口8101)
- rich-code-weaver-file: 文件上传与管理服务(端口8103)
- rich-code-weaver-user: 用户管理服务(端口8102)
- rich-code-weaver-social: 社交互动服务(端口8104)
- rich-code-weaver-prompt: 提示词管理服务(端口8105)

技术栈: Spring Boot 3.x + JDK 21虚拟线程 + Dubbo + Nacos + Redis + MySQL + PostgreSQL(pgvector)

### 1.2 现有异步机制盘点

系统已实现部分异步优化:

**1. JDK 21虚拟线程异步**
- 产物截图生成: `Thread.ofVirtual().start()` (AppServiceImpl.generateAppScreenshotAsync)
- 工作流执行: `Thread.startVirtualThread()` (CodeGenWorkflowApp.executeWorkflow)
- Vue项目构建: `Thread.ofVirtual().name("vue-builder-xxx").start()` (BuildWebProjectExecutor.buildProjectAsync)

**2. CompletableFuture并行执行**
- 资源收集节点: 网络资源整理与图片资源收集并行 (ResourceCollectorNode)
- 命令执行流读取: 异步读取进程输出流与错误流 (ExecuteSysCommandUtil)

**3. SSE流式响应**
- AI代码生成流式输出
- 断线重连机制(基于StreamSession缓存)
- 事件队列缓存(支持lastEventId续传)

### 1.3 核心痛点识别

**痛点1: 长耗时任务阻塞主流程**

虽然已使用虚拟线程异步,但以下场景仍存在隐性阻塞:
- Vue项目构建(npm install + npm run build): 耗时60-300秒
- AI代码生成(工作流/Agent模式): 耗时10-120秒
- 产物截图生成(Playwright截图): 耗时3-10秒
- 文件上传OSS: 耗时1-5秒(取决于文件大小)

**痛点2: 任务状态追踪困难**
- 虚拟线程异步执行后,调用方无法感知任务进度
- 构建失败时用户需刷新页面才能获取错误信息
- 缺乏统一的任务状态管理机制

**痛点3: 并发峰值冲击**
- 多用户同时触发构建时,虚拟线程数量激增
- npm install并发执行可能导致网络带宽瓶颈
- AI模型调用并发受限于API配额

**痛点4: 任务重复执行风险**
- 用户重复点击"部署"按钮可能触发多次构建
- 虽有REFRESHING_APP_IDS防重机制,但仅限单机内存
- 分布式部署下无法跨节点防重

**痛点5: 前端交互割裂**
- Agent模式调用工具时前端显示加载动画
- 若改为消息队列异步,前端需轮询或WebSocket推送
- 需保持现有流式交互体验

## 二、改造目标与原则

### 2.1 改造目标

1. 流量控制: 通过消息队列削峰填谷,避免并发峰值冲击
2. 异步解耦: 长耗时任务与主流程解耦,提升响应速度
3. 可靠性增强: 任务持久化、失败重试、状态可追溯
4. 水平扩展: 支持多消费者并行处理,提升吞吐量
5. 前端体验: 保持流式交互,增强任务进度可视化

### 2.2 改造原则

**原则1: 渐进式改造**
- 优先改造高价值、低风险场景
- 保留现有虚拟线程机制作为降级方案
- 分阶段验证,避免大规模重构风险

**原则2: 前端兼容优先**
- 保持SSE流式响应接口不变
- 增强WebSocket推送作为补充
- 任务状态查询接口向下兼容

**原则3: 健壮性设计**
- 消息持久化(RabbitMQ持久化队列)
- 失败重试机制(指数退避策略)
- 死信队列处理(DLX)
- 幂等性保证(基于Redis分布式锁)

**原则4: 可观测性**
- 任务全生命周期日志
- Prometheus指标采集(队列深度、消费延迟、失败率)
- Grafana可视化监控

## 三、技术选型

### 3.1 消息队列选型

**推荐方案: RabbitMQ**

选型理由:
1. 成熟稳定: 企业级消息队列,社区活跃
2. 功能丰富: 支持延迟队列、死信队列、优先级队列
3. 运维友好: 管理界面完善,监控指标丰富
4. Spring集成: Spring AMQP无缝集成
5. 资源占用: 相比Kafka更轻量,适合中小规模场景

备选方案对比:
- Kafka: 吞吐量高但运维复杂,本项目暂不需要百万级TPS
- RocketMQ: 阿里开源,但社区活跃度不如RabbitMQ
- Redis Stream: 轻量但可靠性不如专业MQ

### 3.2 任务状态管理

**方案: Redis + MySQL双写**


- Redis: 存储任务实时状态(进行中/成功/失败),支持快速查询
- MySQL: 持久化任务历史记录,支持审计与统计分析
- 数据模型: task_execution表(id, task_type, app_id, user_id, status, progress, result, create_time, update_time)

### 3.3 前端通信方案

**方案: SSE + WebSocket混合模式**

- SSE: 保留现有AI生成流式输出(单向推送,兼容性好)
- WebSocket: 新增任务状态推送通道(双向通信,实时性强)
- 降级策略: WebSocket连接失败时回退到轮询

## 四、改造方案设计

### 4.1 任务分类与改造策略

根据业务特性,将任务分为三类:

#### 4.1.1 强建议改造(高价值场景)

**场景1: Vue项目构建任务**

当前实现:
```java
// BuildWebProjectExecutor.buildProjectAsync
Thread.ofVirtual().name("vue-builder-xxx").start(() -> {
    buildProject(projectPath);
});
```

改造方案:
- 发送消息: 部署/刷新接口发送BUILD_VUE_PROJECT消息到队列
- 消费处理: VueProjectBuildConsumer消费消息,执行构建
- 状态推送: 构建进度通过WebSocket推送到前端
- 结果通知: 构建完成后更新App表deployed_time字段

改造价值:
- 削峰填谷: 限制并发构建数量(如最多5个并行)
- 失败重试: npm install失败自动重试3次
- 进度可视化: 前端实时显示"依赖安装中(30%)→构建中(60%)→完成(100%)"

技术要点:
- 队列配置: durable=true, prefetch=1(单消费者串行)
- 幂等性: 基于appId+buildVersion的Redis分布式锁
- 超时控制: 消息TTL=600秒,超时自动进入死信队列

**场景2: 产物截图生成任务**

当前实现:
```java
// AppServiceImpl.generateAppScreenshotAsync
Thread.ofVirtual().start(() -> {
    String screenshotUrl = screenshotService.generateAndUploadScreenshot(appUrl);
    updateApp.setCover(screenshotUrl);
});
```

改造方案:
- 发送消息: 部署成功后发送GENERATE_SCREENSHOT消息
- 消费处理: ScreenshotConsumer调用Playwright截图并上传OSS
- 异步更新: 截图完成后更新App表cover字段
- 失败降级: 3次重试失败后使用默认封面

改造价值:
- 解耦部署流程: 部署接口立即返回,不等待截图完成
- 并发控制: 限制Playwright实例数量(避免内存溢出)
- 失败容错: 截图失败不影响部署成功


**场景3: 文件上传OSS任务**

当前实现:
```java
// FileServiceImpl.upload (同步阻塞)
ossClient.putObject(bucketName, uploadFileName, file.getInputStream());
```

改造方案:
- 发送消息: 文件先存储到本地临时目录,发送UPLOAD_FILE消息
- 消费处理: FileUploadConsumer异步上传到OSS
- 回调通知: 上传完成后通过WebSocket通知前端
- 临时文件清理: 上传成功后删除本地文件

改造价值:
- 接口响应加速: 上传接口从5秒降至50毫秒
- 失败重试: 网络抖动时自动重试
- 批量优化: 支持批量上传合并(未来扩展)

#### 4.1.2 建议保留(低改造价值)

**场景4: 资源收集节点并行任务**

当前实现:
```java
// ResourceCollectorNode (CompletableFuture并行)
CompletableFuture<String> webFuture = CompletableFuture.supplyAsync(() -> {...});
CompletableFuture<String> imageFuture = CompletableFuture.supplyAsync(() -> {...});
CompletableFuture.allOf(webFuture, imageFuture).join();
```

保留理由:
- 执行时间短: 单次调用1-3秒,无明显阻塞
- 强依赖关系: 工作流下一步需等待资源收集完成
- 改造成本高: 需拆分工作流状态机,复杂度激增

优化建议:
- 保持现有CompletableFuture并行机制
- 增加超时控制(如10秒超时)
- 失败降级(资源收集失败时使用空结果继续)

**场景5: AI代码生成流式输出**

当前实现:
```java
// CodeGenWorkflowApp.executeWorkflow (SSE流式)
Flux<ServerSentEvent<String>> generationFlux = ...;
generationFlux.subscribe(...);
return createFollowFlux(sessionKey, lastEventId);
```

保留理由:
- 用户体验核心: 流式输出是产品核心交互
- 已有断线重连: StreamSession机制已解决可靠性
- 改造风险高: 改为消息队列会破坏流式体验

优化建议:
- 保持SSE流式响应
- 增强StreamSession持久化(当前仅内存缓存)
- 考虑Redis Pub/Sub实现跨节点流式推送

#### 4.1.3 不建议改造(高风险场景)

**场景6: Agent工具调用**

当前实现:
- Agent模式通过ToolCallAgent同步调用工具
- 前端显示工具调用加载动画
- 工具执行结果直接返回给AI继续推理

不改造理由:
- 强同步依赖: AI需等待工具结果才能继续
- 交互复杂度: 异步化需重构整个Agent推理循环
- 前端适配成本: 需重新设计加载状态机

**场景7: 命令执行流读取**

当前实现:
```java
// ExecuteSysCommandUtil (CompletableFuture异步读取)
CompletableFuture<String> outputFuture = readStream(process.getInputStream());
CompletableFuture<String> errorFuture = readStream(process.getErrorStream());
```

不改造理由:
- 底层工具方法: 被多处调用,改造影响面大
- 性能已优化: CompletableFuture并行读取已足够高效
- 无业务价值: 改为消息队列无明显收益

### 5.1 消息队列架构设计

#### 5.1.1 队列拓扑结构

### 5.2 核心代码实现

#### 5.2.1 消息生产者

**VueProjectBuildProducer.java**
```java
@Component
@Slf4j
public class VueProjectBuildProducer {
    
    @Resource
    private RabbitTemplate rabbitTemplate;
    
    @Resource
    private RedisTemplate<String, String> redisTemplate;
    
    private static final String BUILD_EXCHANGE = "build.exchange";
    private static final String BUILD_ROUTING_KEY = "build.vue.project";
    private static final String BUILD_LOCK_PREFIX = "build:lock:";
    
    public boolean sendBuildTask(Long appId, String projectPath, String deployKey) {
        // 1. 幂等性检查(分布式锁)
        String lockKey = BUILD_LOCK_PREFIX + appId;
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", Duration.ofMinutes(10));
        
        if (Boolean.FALSE.equals(locked)) {
            log.warn("产物{}正在构建中,跳过重复任务", appId);
            return false;
        }
        
        try {
            // 2. 构建消息体
            BuildTaskMessage message = BuildTaskMessage.builder()
                .messageId(UUID.randomUUID().toString())
                .taskType("BUILD_VUE_PROJECT")
                .appId(appId)
                .payload(Map.of(
                    "projectPath", projectPath,
                    "buildVersion", LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyyMMddHHmmss")),
                    "deployKey", deployKey
                ))
                .retryCount(0)
                .maxRetries(3)
                .createTime(LocalDateTime.now())
                .traceId(MDC.get("traceId"))
                .build();
            
            // 3. 发送消息
            rabbitTemplate.convertAndSend(BUILD_EXCHANGE, BUILD_ROUTING_KEY, message);
            
            // 4. 初始化任务状态
            TaskExecution taskExecution = new TaskExecution();
            taskExecution.setTaskType("BUILD_VUE_PROJECT");
            taskExecution.setAppId(appId);
            taskExecution.setStatus("PENDING");
            taskExecution.setProgress(0);
            taskExecutionService.save(taskExecution);
            
            log.info("发送Vue构建任务: appId={}, messageId={}", appId, message.getMessageId());
            return true;
        } catch (Exception e) {
            // 发送失败释放锁
            redisTemplate.delete(lockKey);
            log.error("发送Vue构建任务失败: appId={}", appId, e);
            throw new BusinessException(ErrorCode.SYSTEM_ERROR, "构建任务发送失败");
        }
    }
}
```


#### 5.2.2 消息消费者

**VueProjectBuildConsumer.java**
```java
@Component
@Slf4j
public class VueProjectBuildConsumer {
    
    @Resource
    private BuildWebProjectExecutor buildExecutor;
    
    @Resource
    private TaskExecutionService taskExecutionService;
    
    @Resource
    private WebSocketTaskNotifier webSocketNotifier;
    
    @Resource
    private RedisTemplate<String, String> redisTemplate;
    
    @RabbitListener(queues = "build.vue.project.queue")
    public void consumeBuildTask(BuildTaskMessage message, Channel channel, 
                                  @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag) {
        String lockKey = "build:lock:" + message.getAppId();
        
        try {
            log.info("开始处理Vue构建任务: appId={}, messageId={}", 
                message.getAppId(), message.getMessageId());
            
            // 1. 更新任务状态为执行中
            taskExecutionService.updateStatus(message.getAppId(), "RUNNING", 0);
            webSocketNotifier.notifyProgress(message.getAppId(), "RUNNING", 0, "开始构建");
            
            // 2. 执行npm install
            String projectPath = (String) message.getPayload().get("projectPath");
            taskExecutionService.updateStatus(message.getAppId(), "RUNNING", 30);
            webSocketNotifier.notifyProgress(message.getAppId(), "RUNNING", 30, "安装依赖中");
            
            ExecuteSysCommandUtil.CommandResult installResult = 
                ExecuteSysCommandUtil.executeCommandWithLog(
                    new File(projectPath), "npm install", 300);
            
            if (!installResult.success()) {
                throw new BusinessException(ErrorCode.OPERATION_ERROR, 
                    "依赖安装失败: " + installResult.log());
            }
            
            // 3. 执行npm run build
            taskExecutionService.updateStatus(message.getAppId(), "RUNNING", 60);
            webSocketNotifier.notifyProgress(message.getAppId(), "RUNNING", 60, "构建项目中");
            
            ExecuteSysCommandUtil.CommandResult buildResult = 
                ExecuteSysCommandUtil.executeCommandWithLog(
                    new File(projectPath), "npm run build", 300);
            
            if (!buildResult.success()) {
                throw new BusinessException(ErrorCode.OPERATION_ERROR, 
                    "项目构建失败: " + buildResult.log());
            }
            
            // 4. 验证dist目录
            File distDir = new File(projectPath, "dist");
            if (!distDir.exists()) {
                throw new BusinessException(ErrorCode.OPERATION_ERROR, "dist目录未生成");
            }
            
            // 5. 更新任务状态为成功
            taskExecutionService.updateStatus(message.getAppId(), "SUCCESS", 100);
            webSocketNotifier.notifyProgress(message.getAppId(), "SUCCESS", 100, "构建完成");
            
            // 6. 手动ACK
            channel.basicAck(deliveryTag, false);
            
            log.info("Vue构建任务完成: appId={}", message.getAppId());
            
        } catch (Exception e) {
            log.error("Vue构建任务失败: appId={}, error={}", message.getAppId(), e.getMessage(), e);
            
            // 失败处理
            handleBuildFailure(message, channel, deliveryTag, e);
        } finally {
            // 释放分布式锁
            redisTemplate.delete(lockKey);
        }
    }
    
    private void handleBuildFailure(BuildTaskMessage message, Channel channel, 
                                     long deliveryTag, Exception e) {
        try {
            int retryCount = message.getRetryCount();
            int maxRetries = message.getMaxRetries();
            
            if (retryCount < maxRetries) {
                // 重试: NACK并重新入队
                log.warn("构建失败,准备重试: appId={}, retryCount={}/{}", 
                    message.getAppId(), retryCount + 1, maxRetries);
                
                message.setRetryCount(retryCount + 1);
                taskExecutionService.updateStatus(message.getAppId(), "RETRYING", 0);
                
                // 延迟重试(指数退避)
                int delaySeconds = (int) Math.pow(2, retryCount) * 10; // 10s, 20s, 40s
                channel.basicNack(deliveryTag, false, false); // 进入死信队列
                
            } else {
                // 超过最大重试次数,标记为失败
                log.error("构建失败且已达最大重试次数: appId={}", message.getAppId());
                
                taskExecutionService.updateStatus(message.getAppId(), "FAILED", 0);
                taskExecutionService.updateResult(message.getAppId(), e.getMessage());
                webSocketNotifier.notifyProgress(message.getAppId(), "FAILED", 0, 
                    "构建失败: " + e.getMessage());
                
                channel.basicAck(deliveryTag, false); // 确认消费,不再重试
            }
        } catch (IOException ioException) {
            log.error("消息ACK/NACK失败", ioException);
        }
    }
}
```


#### 5.2.3 WebSocket任务通知

**WebSocketTaskNotifier.java**
```java
@Component
@Slf4j
public class WebSocketTaskNotifier {
    
    @Resource
    private SimpMessagingTemplate messagingTemplate;
    
    public void notifyProgress(Long appId, String status, int progress, String message) {
        TaskProgressMessage progressMessage = TaskProgressMessage.builder()
            .appId(appId)
            .status(status)
            .progress(progress)
            .message(message)
            .timestamp(System.currentTimeMillis())
            .build();
        
        // 推送到指定产物的订阅者
        String destination = "/topic/task/progress/" + appId;
        messagingTemplate.convertAndSend(destination, progressMessage);
        
        log.debug("推送任务进度: appId={}, status={}, progress={}%", appId, status, progress);
    }
}
```

**WebSocket配置**
```java
@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {
    
    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
```

```javascript
// taskProgressService.js
import SockJS from 'sockjs-client';
import Stomp from 'stompjs';

class TaskProgressService {
  constructor() {
    this.stompClient = null;
    this.subscriptions = new Map();
  }
  
  connect() {
    const socket = new SockJS('/ws/task');
    this.stompClient = Stomp.over(socket);
    
    this.stompClient.connect({}, (frame) => {
      console.log('WebSocket连接成功:', frame);
    }, (error) => {
      console.error('WebSocket连接失败:', error);
      // 降级到轮询
      this.fallbackToPolling();
    });
  }
  
  subscribeTaskProgress(appId, callback) {
    if (!this.stompClient || !this.stompClient.connected) {
      this.connect();
    }
    
    const destination = `/topic/task/progress/${appId}`;
    const subscription = this.stompClient.subscribe(destination, (message) => {
      const progress = JSON.parse(message.body);
      callback(progress);
    });
    
    this.subscriptions.set(appId, subscription);
  }
  
  unsubscribeTaskProgress(appId) {
    const subscription = this.subscriptions.get(appId);
    if (subscription) {
      subscription.unsubscribe();
      this.subscriptions.delete(appId);
    }
  }
  
  fallbackToPolling() {
    // 降级到轮询方案
    console.warn('WebSocket不可用,降级到轮询模式');
    // 实现轮询逻辑...
  }
}

export default new TaskProgressService();
```

### 6.2 风险控制

**风险1: 消息丢失**
- 缓解措施: RabbitMQ持久化队列 + 消息持久化 + 手动ACK
- 降级方案: 消息发送失败时回退到虚拟线程同步执行

**风险2: 消费者宕机**
- 缓解措施: 消息未ACK自动重新入队 + 多消费者实例
- 降级方案: 监控告警 + 自动重启消费者

**风险3: 队列积压**
- 缓解措施: 限流(令牌桶) + 队列深度告警 + 动态扩容消费者
- 降级方案: 拒绝新任务提交 + 提示用户稍后重试

**风险4: WebSocket连接失败**
- 缓解措施: 自动重连 + 降级到轮询
- 降级方案: 前端主动查询任务状态接口

**风险5: 数据库写入失败**
- 缓解措施: 事务保证 + 失败重试
- 降级方案: 仅更新Redis缓存,异步补偿写入数据库

### 6.3 回滚方案

**快速回滚步骤**
1. 停止消息生产者(注释发送消息代码)
2. 恢复原有虚拟线程异步逻辑
3. 清空RabbitMQ队列(避免积压消息)
4. 重启服务

**数据一致性保证**
- 回滚前确保队列中消息已全部消费完成
- 检查task_execution表中无RUNNING状态任务
- 清理Redis中残留的分布式锁

## 八、最佳实践建议

### 8.1 消息队列使用规范

1. 消息幂等性: 所有消费者必须实现幂等性(基于messageId去重)

### 8.2 任务状态管理规范

1. 状态机设计: PENDING → RUNNING → SUCCESS/FAILED/RETRYING
2. 进度更新: 关键节点更新进度(0% → 30% → 60% → 100%)
3. 结果记录: 成功时记录输出路径,失败时记录错误信息
4. 历史清理: 定期清理30天前的历史记录(保留统计数据)

### 8.3 前端交互规范

1. 加载状态: 任务提交后立即显示"任务已提交,正在处理中"
2. 进度展示: 通过进度条实时展示任务进度
3. 结果通知: 任务完成后弹窗通知用户
4. 错误处理: 失败时展示详细错误信息,提供重试按钮

```java
@Configuration
public class RabbitMQConfig {
    
    @Bean
    public Queue buildVueProjectQueue() {
        return QueueBuilder.durable("build.vue.project.queue")
            .withArgument("x-message-ttl", 600000) // 10分钟TTL
            .withArgument("x-dead-letter-exchange", "dlx.exchange")
            .withArgument("x-dead-letter-routing-key", "dlx")
            .build();
    }
    
    @Bean
    public DirectExchange buildExchange() {
        return new DirectExchange("build.exchange", true, false);
    }
    
    @Bean
    public Binding buildBinding() {
        return BindingBuilder.bind(buildVueProjectQueue())
            .to(buildExchange())
            .with("build.vue.project");
    }
    
    @Bean
    public RabbitTemplate rabbitTemplate(ConnectionFactory connectionFactory) {
        RabbitTemplate template = new RabbitTemplate(connectionFactory);
        template.setMessageConverter(new Jackson2JsonMessageConverter());
        template.setMandatory(true);
        template.setConfirmCallback((correlationData, ack, cause) -> {
            if (!ack) {
                log.error("消息发送失败: {}", cause);
            }
        });
        return template;
    }
```
